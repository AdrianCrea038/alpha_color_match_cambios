          MANUAL DE FUNCIONAMIENTO - ALPHA COLOR MATCH 2 (V4.3)
================================================================================

Este documento describe la arquitectura, lógica y flujo de datos del sistema
Alpha Color Match 2 para referencia técnica y futuras modificaciones.

1. PROPÓSITO DEL SISTEMA
------------------------
Alpha Color Match 2 es una suite profesional para la gestión de desarrollo de
colores, validación de archivos .txt maestros (NK) y monitoreo de producción
en planta a través de un Dashboard Ejecutivo de alto rendimiento.

2. ARQUITECTURA TÉCNICA
-----------------------
- Frontend: HTML5, CSS3 (Premium Dark Mode / Glassmorphism) y JS Vanilla (ES Modules).
- Backend: Supabase (PostgreSQL) con conexión atómica vía API.
- Paleta de Colores CSS (Identidad Visual):
  * Primario (Acción): Azul Eléctrico (#3b82f6) - Sombras: 0 4px 15px rgba(59, 130, 246, 0.3)
  * Éxito (Válido): Verde Esmeralda (#10b981)
  * Error Crítico: Rojo Neón (#ef4444)
  * Alerta/Catálogo: Ámbar (#fbbf24) y Naranja Vivo (#f97316)
  * Informativo: Azul Cielo (#60a5fa)
  * Fondo: Slate Dark (#0f172a / #1e293b) con bordes de 1px solid rgba(255,255,255,0.05).

3. MOTOR DE VALIDACIÓN Y AUDITORÍA (V4.3)
---------------------------------------
Archivo: js/modules/nameValidator.js (El Corazón del Sistema)

A. REGLAS DE FILTRADO (IGNORAR):
   - Todo registro que contenga la palabra "WHITE" en su nombre es ignorado por el 
     motor de auditoría (se considera válido por defecto).
   - Formatos Tonal: Colores que inician con "TN", "TNL" o "TONAL" no requieren NK.

B. VALIDACIÓN UNIFICADA DE NOMBRES:
   - Un color es válido si existe en el catálogo de Equivalencias bajo su 
     Nombre Base (ej: "002 BLACK") O bajo su Nombre Completo (ej: "002 BLACK NK711075").

C. GESTIÓN DE BASE DE DATOS DESDE AUDITORÍA:
   - Agregar a Grupo: Inserta el nombre en la columna 'colores' (text[]) de la tabla 
     'equivalencias' en Supabase.
   - Crear Nuevo Grupo: Permite crear una fila nueva en 'equivalencias' con un ID 
     de grupo personalizado (ej: un nuevo NK).
   - Detección Dinámica: El sistema detecta automáticamente si la columna de ID se 
     llama 'nk', 'grupo_id', 'nk_code', etc., para evitar errores de esquema.

4. MÓDULOS ESPECÍFICOS Y SU PROGRAMACIÓN
----------------------------------------

   A. AUDITORÍA DIRECTA (js/views/directAuditView.js)
      - Sincronización Fresh: Al cargar un archivo, ejecuta ensureValidColorCatalogLoaded(true) 
        para bajar los últimos cambios de Supabase (ignora el caché).
      - Visualización: Filtra por 7 estados: Catálogo, CMYK Diferente, CMYK > 100, 
        Sin NK, NK Inválido, Duplicados y Paréntesis.

   B. COMPARADOR PRINCIPAL (js/main.js & js/modules/resultsRenderer.js)
      - Reconciliación: Compara Master vs Secundario y asigna estados de color (Verde/Rojo/Naranja).
      - Bulk Actions: addAllPendingItems() agrega todos los cambios propuestos de una sola vez.

   C. MANEJO DE DUPLICADOS (js/modules/duplicateHandler.js)
      - Firma Única: La clave de duplicado es stricta: "NombreBase | NK".

5. REGLAS DE ORO PARA LA IA
----------------------------
1. NUNCA modificar la tabla 'master_nks' durante una auditoría de colores (Nombres).
2. CMYK > 100: Es un error CORRUPTO. Bloquea exportación y se marca en ROJO NEÓN.
3. FORMATO ARRAY: Al escribir en la columna 'colores' de Supabase, usar siempre el 
   formato de Postgres Array: {"NOMBRE"}.
4. NORMALIZACIÓN: Antes de comparar con mapas de constantes, limpiar el nombre 
   quitando espacios y caracteres especiales.
